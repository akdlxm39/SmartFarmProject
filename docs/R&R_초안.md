# SmartFarmProject R&R 초안

작성일: 2026-06-22
기준: 3인 프로젝트 / 시뮬레이션과 현실 구현을 병행 관리

## 1. 역할 배분 원칙
이 프로젝트는 크게 아래 3개 축으로 나눠 운영하는 것이 가장 안정적이다.

1. **비전/임베디드 축**
2. **로봇/시뮬레이션/자율주행 축**
3. **서버/프론트/통합 문서화 축**

이렇게 나누는 이유는 다음과 같다.
- 1번 카메라 판정 로직과 2번 카메라 흐름 확인 로직은 데이터/센서 중심 작업이다.
- Dobot, RoboDK, TurtleBot, SLAM은 로봇/시뮬레이션 중심 작업이다.
- Django, Vue, DB, 대시보드, 통합 로그는 서버/화면/정리 중심 작업이다.

---

## 2. 1차 R&R 초안

### Participant1 — 비전/임베디드 리드
**주 담당 영역**
- 1번 카메라 비전 파이프라인
- 2번 카메라(RGB-D) 흐름 감지
- 라즈베리파이 연동
- 데이터셋 수집/라벨링/학습

**핵심 책임**
- 작물 3종 분류 + 불량 판별 모델 설계/학습
- 1번 카메라 3방향 촬영 입력 구조 정의
- 2번 카메라의 존재/위치/흐름 확인 로직 구현
- 비전 결과를 ROS/서버로 넘길 데이터 포맷 정의 지원
- 라즈베리파이 추론 가능 여부 검증

**주요 산출물**
- 데이터셋 구조 및 라벨링 기준
- YOLO 학습 결과
- 추론 파이프라인 코드
- 카메라별 입출력 명세

**의존 관계**
- Dobot 촬영 포즈는 Participant2와 협의 필요
- 서버 로그 포맷은 Participant3와 협의 필요

---

### Participant2 — 로봇/시뮬레이션/자율주행 리드
**주 담당 영역**
- Dobot 제어
- RoboDK 시뮬레이션
- TurtleBot / SLAM / 자율주행
- 물리 동선 및 적재/폐기 구조

**핵심 책임**
- Dobot 수확 -> 카메라 -> 적재/폐기 동작 흐름 구현
- 시뮬레이션에서 3열 분류 구조 유지
- 현실 구현에서는 불량 즉시 분리 / 정상 컨베이어 적재 구조 구현
- TurtleBot SLAM, 맵핑, 목표 지점 이동 구현
- 컨베이어 끝 정상 수거 구조와 불량품 상자 위치 협의

**주요 산출물**
- RoboDK 시뮬레이션 파일/결과
- Dobot 동작 시퀀스
- TurtleBot 맵 및 주행 데모
- 물리 배치도 초안

**의존 관계**
- 비전 판정 완료 신호는 Participant1과 인터페이스 합의 필요
- 상태 로그/대시보드 연동은 Participant3와 협의 필요

---

### Participant3 — 서버/프론트/통합 리드
**주 담당 영역**
- Django / MySQL
- Vue / Chart.js 대시보드
- ROS-서버 브리지
- Git/문서/시연 자료 통합

**핵심 책임**
- Django 서버 초기 세팅 및 DB 설계
- 수확 로그 / 비전 판정 결과 / 장치 상태 저장 구조 설계
- Vue 기반 대시보드 구축
- ROS Bridge 또는 대체 연동 구조 정리
- README / docs / 발표자료 / 진행 로그 관리

**주요 산출물**
- ERD 초안
- API/로그 데이터 구조
- 웹 대시보드 화면
- 통합 시연 로그 및 발표자료

**의존 관계**
- 비전 결과 포맷은 Participant1과 합의 필요
- 로봇/자율주행 상태값은 Participant2와 합의 필요

## 5. 우선순위 기준

### 1차 핵심(MVP 필수)
- 1번 카메라 판정
- Dobot 수확/폐기/정상 적재
- 2번 카메라 기반 컨베이어 흐름 확인
- TurtleBot SLAM 및 자율주행 기본 성공
- 기본 웹 로그/대시보드

### 2차 고도화
- 라즈베리파이 엣지 최적화
- 통계 고도화
- 물/비료 분사 기능
- 발표용 시뮬레이션 연출 강화

---

## 6. 현재 기준 추천 업무 분할 방식
가장 현실적인 초기 분할은 아래와 같다.

- **Participant1:** AI 모델링 + 카메라 1/2 + 라즈베리파이
- **Participant2:** Dobot + RoboDK + TurtleBot + 물리 배치
- **Participant3:** Django + DB + Vue + 통합 문서/발표

이 구조가 좋은 이유:
- 센서/비전과 로봇 제어가 자연스럽게 분리됨
- TurtleBot을 별도 사람에게 모으지 않고 로봇 축에 묶을 수 있음
- 웹/서버 담당이 통합 로그와 발표 정리를 함께 가져갈 수 있음
