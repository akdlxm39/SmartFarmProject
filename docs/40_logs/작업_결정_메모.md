Dobot 흡착 대기, 촬영 신호, HOME/좌표 보정, 안전 이동 방식

- Dobot 좌표표는 현재 기준 좌표로 유지하되, 최종 장비 배치 후 현재 pose 읽기 기반으로 갱신해야 한다.
- 다음 구현 단위는 `harvest_index -> jump move -> 석션 2초 대기 -> 3방향 촬영 -> 임의 정상/불량 분기` 테스트가 된다.
- 아직 정할 값은 jump move 안전 z 높이와 비전 결과 대기 timeout이다.

- 고정 카메라 + 고정 벨트에서는 먼저 ROI 기반 OpenCV 감지로 `entered -> moving -> exited/delivered` 이벤트를 안정화하는 것이 가장 빠른 검증 경로다.

## 결정 주제
컨베이어 RealSense 단일 ROI 큐브 감지와 Modbus 제어 방향

## 현재 결정
- D435i는 ROS 2 `realsense2_camera` wrapper를 통해 color/depth topic을 받는다.
- 감지 대상은 컨베이어 위 **빨간색/녹색 큐브**다.
- 프레임이 비스듬하게 찍히므로, 실제 detector와 같은 ROS color topic frame을 직접 받은 뒤 컨베이어 평면 4점을 클릭해 top-view로 보정하고, 보정 프레임 기준으로 `conveyor_roi`를 지정한다.
- 좌표 세팅 코드는 `/home/ssafy/work/SmartFarmProject/workspaces/지웅/conveyor/scripts/select_conveyor_roi.py`에서 관리하고, `--ros-topic /camera/camera/color/image_raw` 입력을 권장하며, 결과는 `config/conveyor_roi_topview.json`에 저장한다.
- top-view 출력 크기는 기본적으로 원본 ROS frame 크기와 같은 `--topview-size-mode raw`를 사용한다. 클릭한 quad 자체 크기(`quad` 모드)를 쓰면 top-view 창이 너무 작아질 수 있기 때문이다.

## 결정 주제
컨베이어 Modbus 비동기화와 Raspberry Pi 5 제어 역할 분리

## 현재 결정
- 기존 `pymodbus==3.9.2` 동기식 client 기반 구조는 Modbus read/write 시 버벅임을 유발할 수 있으므로, **`pymodbus==3.13.1`** 기준으로 전환한다.
- Modbus TCP server는 `192.168.110.109:50200`의 외부/shared register layer로 둔다. Raspberry Pi 5/Raspbian은 **Modbus client + 컨베이어 모터 GPIO controller**를 담당한다.
- PC/ROS2 D435i/OpenCV 노드는 Modbus client로서 비전 처리와 command/vision register write를 담당하되, `AsyncModbusTcpClient` 기반 async worker를 사용해 image callback에서 네트워크 I/O를 기다리지 않는다.
- Pi는 실제 모터 상태 기준으로 `40023 conveyor_status`, `40024 conveyor_error_code`를 갱신한다.
- PC는 `40021 conveyor_command`, `40022 conveyor_speed_cmd`, `40025 cube_detected`, `40026 cube_color`, `40027 last_vision_event` 중심으로 쓴다.

## 이유
- D435i frame 처리, OpenCV ROI 감지, Modbus read/write가 같은 동기 흐름에 묶이면 실시간성이 떨어진다.
- Raspberry Pi가 컨베이어 모터와 가까운 위치에서 실제 모터 상태 owner가 되고, 외부 Modbus server가 shared register layer를 유지하면 모터 상태와 register 상태를 분리해 관리하기 쉽다.
- `pymodbus==3.13.1`의 async client API를 사용하면 여러 client 접근과 네트워크 지연이 ROS image callback을 직접 막지 않게 설계할 수 있다.

## 영향
- 기존 ROS2 `conveyor_modbus.py`는 동기 `ModbusTcpClient` wrapper에서 async writer 구조로 바뀐다.
- Raspberry Pi 전용 코드는 `workspaces/지웅/conveyor/pi_controller/` 아래에 분리한다.
- 상세 전환 계획은 `docs/Conveyor_Modbus_Async_RaspberryPi_전환_계획.md`와 Obsidian `[[Conveyor Modbus Async + Raspberry Pi 전환 계획]]`에 기록한다.

## 결정 주제
컨베이어 RealSense ROS 비전 노드의 Modbus command gating 정책

## 현재 결정
- PC/ROS2 D435i/OpenCV 비전 노드는 큐브 탐지 상태에 따라 `40021 conveyor_command`, `40022 conveyor_speed_cmd`, `40025 cube_detected`, `40026 cube_color`, `40027 last_vision_event`를 갱신한다.
- PC/ROS2 비전 노드는 `40023 conveyor_status`, `40024 conveyor_error_code`를 쓰지 않는다. 두 register는 Raspberry Pi controller가 실제 모터/버튼 상태 기준으로 소유한다.
- 비전 노드는 command write 직전에 `40023/40024`를 읽는다.
- Pi status가 `4 emergency_stopped`이면 비전 노드는 **어떤 command/speed도 쓰지 않고**, vision register만 갱신한다.
- Pi status가 `3 error`이면 비전 노드는 run/reset을 내리지 않는다. 단, stable disappearance 등으로 stop이 필요한 경우 `stop`만 허용한다.
- Pi status가 `0 idle`, `1 running`, `2 delivered`이면 ROI state machine 결과에 따라 run/stop을 허용한다.
- status read 실패 시에는 emergency 여부를 확신할 수 없으므로 fail-safe로 command 없이 vision register만 쓴다.

## 이유
- Pi의 물리 긴급정지 버튼은 command register를 바꾸지 않고 `40023/40024`로 상태를 표현하기로 했다. 따라서 PC 비전 노드는 Pi-owned status를 먼저 읽어 command를 gate해야 한다.
- PC가 `40023/40024`를 덮어쓰면 Pi의 실제 모터 상태와 register 상태가 충돌한다.
- status read 실패를 run 허용으로 처리하면 네트워크/서버 이상 시 예기치 않게 컨베이어를 움직일 수 있으므로, command를 막는 쪽이 안전하다.

## 영향
- `workspaces/지웅/ros2_ws/src/conveyor_vision_test/conveyor_modbus.py`에 `ConveyorPhysicalState`, `ConveyorCommandWritePlan`, `build_command_write_plan()`, `read_physical_state()`, `write_command_plan()`을 추가했다.
- `topview_color_detector`는 기존 full block write 대신 status read → command plan 생성 → command/vision register 분리 write를 사용한다.
- 실제 server 통합 테스트에서는 Pi local emergency 상태에서 `40021/40022`가 바뀌지 않는지 반드시 확인한다.
