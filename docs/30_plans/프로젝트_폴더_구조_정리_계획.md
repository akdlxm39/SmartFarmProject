## 3. 현재 구조의 문제점

1. **docs가 너무 평면적이다.**
   - 설계 문서, 작업 메모, 구현 계획, 다이어그램 설명이 한 폴더에 섞여 있다.

2. **workspaces는 초기 협업에는 좋지만, 통합 개발에는 불리하다.**
   - 각자 폴더에 구현물이 흩어져 있어 최종 실행 경로가 보이지 않는다.

3. **실행 코드와 실험 산출물이 섞일 위험이 있다.**
   - 특히 이미지, 캘리브레이션 preview, 모델 파일, 테스트 캡처 결과를 어디까지 Git에 둘지 기준이 필요하다.

4. **frontend/backend/robot/vision/modbus 경계가 아직 공통 구조로 드러나지 않는다.**
   - 현재는 팀원별 작업 흔적은 잘 보이지만, 최종 시스템 구조는 README만 보고 파악하기 어렵다.

5. **Git 관리 기준이 더 필요하다.**
   - `.venv`, build/install/log, incoming images/videos 등은 잘 ignore되고 있지만,
   - preview 이미지, 모델 파일(`best.pt`), screenshot, calibration backup JSON 등은 별도 정책이 필요하다.

## 4. 정리 방향 원칙

### 원칙 1. 지금 당장 workspaces를 없애지 않는다
현재 팀원들이 개인 폴더 기준으로 작업하고 있으므로, 바로 이동하면 충돌/혼란이 크다.

따라서 1차 정리는 다음 방식이 좋다.
- `workspaces/`는 **개인 실험/초기 개발 보관소**로 유지
- 통합 가능한 코드만 새 공통 폴더로 복사/승격
- 승격된 뒤에도 원본은 일정 기간 유지

### 원칙 2. 최종 실행 구조를 루트에 따로 만든다
`workspaces/`와 별도로 최종 통합 구조를 만든다.

- `.venv/`, `build/`, `install/`, `log/`, `__pycache__/`
- 카메라 수신 결과 `incoming_jpegs/`, `incoming_videos/`
- 반복 생성되는 ROI preview/self-test 이미지
- 임시 smoke 결과
- 큰 모델 파일 `best.pt`

### 추가 `.gitignore` 후보
```gitignore
# ROS2
**/ros2_ws/build/
**/ros2_ws/install/
```

## 8. 실행 순서 제안

### Phase 0. 백업/기준점 생성
1. 현재 변경사항을 커밋하거나 별도 branch 생성
2. 정리 전 태그 생성: `before-restructure-YYYYMMDD`
3. GitHub main 상태 확인

### Phase 1. 문서 구조 먼저 정리
1. `docs/00_project`, `docs/10_architecture`, `docs/20_subsystems`, `docs/30_plans`, `docs/40_logs`, `docs/diagrams` 생성
2. 문서를 성격별로 이동
3. README 링크 수정
4. Obsidian 링크도 같이 갱신

### Phase 2. 공통 실행 구조 생성
1. `apps/`, `robot/`, `vision/`, `embedded/`, `modbus/`, `config/`, `data/` 생성
2. 각 폴더에 `README.md` 추가
3. 아직 코드는 이동하지 않고 구조만 만든다

### Phase 3. 안전한 코드 승격
1. Modbus server부터 `modbus/shared_server/`로 복사/이동
2. Conveyor Pi controller를 `embedded/conveyor_pi/`로 이동
3. ROS2 Dobot package를 `robot/dobot/`로 이동
4. D435i detector를 `vision/camera2_conveyor/`로 이동
5. Pi Camera/JPG capture를 `vision/camera1_inspection/`으로 이동
6. Frontend/backend prototype을 `apps/`로 이동

### Phase 4. import/path 수정 및 검증
1. Python import 경로 수정
2. shell script 경로 수정
3. ROS2 workspace build 경로 확인
4. 테스트 실행
   - Modbus: `workspaces/지웅/modbus` 기준 테스트를 새 위치에서 재실행
   - Conveyor Pi: `pytest`
   - ROS2 package: `colcon build`, `colcon test`
   - Frontend: `npm install`, `npm run build`

### Phase 5. workspaces 정리
1. 이동 완료된 개인 폴더에는 `MOVED_TO.md` 또는 README 안내 추가
2. 원본 코드는 일정 기간 유지하거나 archive 처리
3. 최종적으로 `workspaces/`는 개인 실험/미통합 코드만 남긴다

## 9. 추천 우선순위

가장 먼저 정리할 순서:
1. `docs/` 정리
2. `modbus/shared_server/` 승격
3. `embedded/conveyor_pi/` 승격
4. `vision/camera2_conveyor/` 승격
5. `robot/dobot/` 승격
6. `vision/camera1_inspection/` + YOLO inference 통합
7. `apps/frontend`, `apps/backend` 승격
8. TurtleBot 폴더는 실제 코드 들어오기 전 skeleton만 생성

이유:
- Modbus는 모든 장치가 공유할 상태 레이어이므로 가장 먼저 공통 위치로 빼는 것이 좋다.
- Conveyor Pi와 D435i는 이미 테스트/문서가 많아 통합 가치가 높다.
- Web은 상대적으로 독립적이라 뒤에서 API 계약에 맞춰 붙여도 된다.

## 10. 보류할 결정
정리 실행 전에 팀과 확인할 것:
1. 폴더명을 영어로 통일할지, 문서만 한글명을 유지할지
2. `best.pt` 모델 파일을 Git에 계속 둘지, Git LFS/Drive/Release asset으로 뺄지
3. `workspaces/` 원본을 언제까지 유지할지
4. 통합 후 ROS2 workspace를 하나로 합칠지, subsystem별로 나눌지
5. backend는 FastAPI 기준으로 갈지, Django 기준으로 갈지 최종 확정

## 11. 내 추천 결론
지금 바로 전체 파일을 이동하기보다, **문서 정리 + 공통 skeleton 생성 + Modbus부터 승격** 순서로 가는 것이 안전하다.

즉, 다음 실행은 아래 정도가 적절하다.

```text
1. 현재 변경사항 커밋
2. docs 재분류
3. 공통 폴더 skeleton 생성
4. modbus/shared_server 이동
5. README 갱신
6. 테스트 후 커밋
```

이렇게 하면 팀원 작업을 망가뜨리지 않으면서도, 프로젝트가 개인 폴더 모음에서 통합 프로젝트 구조로 넘어가기 시작한다.
