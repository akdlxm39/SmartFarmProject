## 1. 이전 다이어그램이 별로였던 이유
이전 Excalidraw는 단순히 컴포넌트를 나열하고 선을 연결한 수준이라, 현재 프로젝트의 실제 구조를 충분히 반영하지 못했다.

문제점:
1. **현실 구현 흐름과 통신 인프라가 섞여 있음**
   - 작물 수확/검사/컨베이어 이송 흐름과 WebSocket/Modbus/ROS2 통신선이 한 장에 섞여 가독성이 낮았다.
2. **Main Hub 내부 역할 구분이 약함**
   - FastAPI, ROS2, Modbus server, Vision daemon, Dashboard bridge의 책임 경계가 명확하지 않았다.
3. **Modbus server 구조가 최신 내용과 다르게 단순화됨**
   - 현재 확정된 구조는 `192.168.110.109:50200` shared register layer이고, PC vision node와 Raspberry Pi는 모두 client다.
   - register ownership(`40021~40030`, `40031~40050`, `40051~40070`)을 보여줘야 한다.
4. **카메라 1/2의 권위 차이가 드러나지 않음**
   - 1번 카메라는 작물 3종 + 불량 판정의 authoritative source다.
   - 2번 D435i는 classifier가 아니라 컨베이어 흐름 확인/예외 감지용이다.
5. **Dobot-비전 연동 방식이 빠짐**
   - Dobot 0/120/-120도 촬영 포즈 도달 → PC가 Pi Camera에 capture 명령 → JPG 수신 → 판정 → Dobot 분기 흐름이 핵심인데 잘 보이지 않았다.
6. **TurtleBot과 Web은 MVP 포함이지만 아직 인터페이스 초안임을 표현하지 못함**
   - 확정/구현 중/예정 상태를 시각적으로 구분해야 한다.

## 2. 보강 목표
새 아키텍처는 발표용으로 바로 설명 가능한 수준이어야 한다.

목표:
- 한 장에서 전체 구조를 보되, 선이 난잡하지 않게 **레이어**를 명확히 나눈다.
- 실제 구현된/구현 중인 경로와 아직 예정인 경로를 구분한다.
- `작업 흐름`, `통신 프로토콜`, `상태 저장`, `장치 제어`를 분리해서 보여준다.
- Modbus register ownership과 ROS2/WebSocket/Socket의 역할 차이를 분명히 한다.

## 3. 새 아키텍처 구성 방식
한 장짜리 복잡한 그림 하나로 끝내지 않고, 아래 4종으로 나눈다.

### 3.1 Architecture Map — 전체 시스템 지도
목적: 발표/README용 대표 그림.

레이어:
1. **UI/관제 레이어**
   - Vue Dashboard
   - 실시간 상태/영상/통계 표시
2. **Main Hub 레이어**
   - FastAPI Backend
   - Vision Services
     - Camera1 Pi JPG capture server / AI inference
     - Camera2 D435i top-view ROI detector
   - ROS2 Core
   - Modbus TCP Server / Shared Register Layer
3. **Edge/Device Client 레이어**
   - Raspberry Pi Camera client
   - Raspberry Pi Conveyor GPIO client
   - Dobot ROS2 client/node
   - TurtleBot ROS2/Modbus client
4. **Physical Process 레이어**
   - 수확 영역
   - 1번 카메라 촬영 위치
   - 불량품 상자
   - 컨베이어 시작점
   - 정상 수거 상자
   - TurtleBot 배송/물류

표현 원칙:
- WebSocket: 파란색
- ROS2 Topic/Service/Action: 분홍색
- Modbus TCP: 초록색
- Raw socket/JPG capture: 주황색
- 예정/미구현: 점선

### 3.2 Runtime Flow — 작물 1개 처리 시퀀스
목적: 실제 동작 순서를 설명.

단계:
1. Dobot harvest command 수신
2. Dobot이 수확 좌표로 이동, 석션 ON
3. Dobot이 1번 카메라 위치로 이동
4. Dobot이 0/120/-120도 촬영 포즈를 순차적으로 만든다
5. PC vision server가 Pi Camera client에 `capture` 명령을 보낸다
6. Pi가 JPG를 촬영해 PC로 전송한다
7. FastAPI 또는 vision inference service가 3장 기준으로 작물 종류/정상·불량 판정
8. 불량이면 Dobot이 불량품 상자에 투입
9. 정상이면 Dobot이 컨베이어 시작점에 적재
10. D435i top-view ROI detector가 빨강/초록 큐브/작물 흐름을 감지
11. PC vision node가 Modbus `40021/40022/40025~40027` write
12. Raspberry Pi conveyor client가 `40021/40022`를 read해 GPIO motor 제어
13. Pi가 실제 상태를 `40023/40024`에 write
14. Backend/Dashboard가 상태를 읽고 UI 업데이트
15. 필요 시 TurtleBot 배송 트리거

### 3.3 Modbus Register Ownership Map — 상태/명령 레지스터 지도
목적: 충돌 방지와 통합 기준 제공.

반드시 표시할 block:
- Conveyor: `40021~40030`
  - PC vision/manual client write: `40021`, `40022`, `40025`, `40026`, `40027`
  - Pi conveyor client write: `40023`, `40024`
- Dobot future: `40031~40050`
- TurtleBot future: `40051~40070`
- System/Farm future: `40071~40100`

중요 규칙:
- 같은 register를 두 주체가 동시에 쓰지 않는다.
- Dashboard/backend는 MVP에서는 read 중심으로 둔다.
- emergency 상태에서는 PC vision node가 `40021/40022`를 쓰지 않는다.

### 3.4 Physical/Simulation Split — 현실 구현과 시뮬레이션 차이
목적: 발표 때 혼동 방지.

현실 구현:
- 1번 카메라가 메인 판정
- 불량은 Dobot이 별도 상자에 즉시 분리
- 정상만 컨베이어 이동
- 컨베이어 끝 공용 상자 낙하
- D435i는 흐름 확인

시뮬레이션:
- 3열 분류 구조 유지
- AGV/TurtleBot 물류 흐름 확장 표현
- 트럭/배송 경로 포함 가능

## 4. 보강된 아키텍처에서 반드시 반영할 최신 구현 내용
### 4.1 Vision / Camera1
- Pi Camera는 PC에서 `capture` 명령을 받으면 JPG 1장을 촬영해 socket으로 전송한다.
- PC는 3방향 이미지를 받아 inference input set으로 묶는다.
- 결과는 `inspection_id`, `crop_type`, `quality_status`, `next_action`으로 정리한다.

### 4.2 Dobot
- ROS2 + `magician_ros2` 기반.
- `harvest_test`, `calibrate_positions`, `calibrate_harvest_positions` 작업 흐름을 반영한다.
- 촬영 포즈는 `0`, `120`, `-120`, 복귀 `0`.
- 현재 좌표 보정 파일은 `dobot_positions_latest.json` 계열이다.

### 4.3 Conveyor / D435i
- D435i는 ROS2 `realsense2_camera` 기반 color topic을 사용한다.
- top-view 보정 후 단일 `conveyor_roi`에서 빨강/초록 HSV detection.
- `visible -> disappeared_stable(10프레임)` 상태 전이로 conveyor stop 판단.
- depth는 MVP 필수가 아니라 보조.

### 4.4 Modbus
- Server endpoint: `192.168.110.109:50200`.
- 기준 라이브러리: `pymodbus==3.13.1`.
- PC/ROS2 vision node와 Raspberry Pi 5는 모두 client.
- Pi endpoint: `ssafy@192.168.110.139`는 문서에 비밀번호 저장 금지.
- Pi client는 GPIO motor control과 `40023/40024` owner.

### 4.5 Web / Dashboard
- 효진 workspace의 Vue 앱을 UI 후보로 반영한다.
- Dashboard는 상태/통계/영상/이벤트 로그를 표시한다.
- FastAPI는 WebSocket broker + API + inference gateway 역할로 설계한다.

### 4.6 TurtleBot
- TurtleBot SLAM/자율주행은 MVP 포함.
- 현재 아키텍처에서는 `future/implementation in progress`로 점선 처리한다.
- Modbus block은 `40051~40070`로 예약한다.
